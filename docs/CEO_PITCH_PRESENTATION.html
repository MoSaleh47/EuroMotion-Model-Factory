🔗 Feedback Loops in Our Model

Our model captures 4 critical feedback loops that drive EV supply chain behavior:

🔄

Reinforcing (R1) — Investment

More revenue → More investment → More capacity → More production → More revenue

📈

Reinforcing (R2) — Market Growth

Higher market share → More orders → Better efficiency → Higher market share

⚖️

Balancing (B1) — Supply Constraint

Low chip supply → Lower production → Backlog grows → Pressure to localize supply

🎯

Balancing (B2) — Inventory Control

High inventory → Reduced production rate → Inventory decreases → Balance restored

💡 Reinforcing loops amplify changes (growth or decline). Balancing loops stabilize the system toward a goal. Our model captures both dynamics.

⚠️ The LLM Non-Determinism Problem

If you ask an LLM to create a System Dynamics model twice, the elements can change each time. How do we control this?

❌ Pure Text Approach

Model defined in natural language

  • Difficult to control
  • Updates affect any part randomly
  • No validation possible
  • Can't parse LLM output reliably

✅ JSON Data Structure

Model defined in structured format

  • Control exactly what context goes to LLM
  • Parse and validate LLM output
  • Integrate changes deterministically
  • Version control & audit trail

💡 Our solution: Ask LLM to generate specific changes (e.g., a new parameter), parse the output, then insert it into our JSON structure programmatically.

📄 JSON Structure — The Key to Control

Model Configuration

{
  "stocks": {
    "inventory": {
      "initial_value": 50000,
      "unit": "units"
    }
  },
  "flows": {
    "production_rate": {
      "mechanism": "production",
      "formula": "capacity * efficiency"
    }
  },
  "parameters": {
    "semiconductor_reliability": {
      "value": 85,
      "min": 50, "max": 100
    }
  },
  "feedback_loops": { ... }
}

Why This Structure Works

🎯 Deterministic Updates

LLM suggests: "Change semiconductor_reliability to 70" → We update exactly one field

✅ Validation

Before applying: check min/max bounds, verify field exists

📝 Audit Trail

Every change tracked, can revert, compare scenarios

🔄 Reusable Scenarios

crisis_semiconductor.json, aggressive_growth.json, etc.

🔌 Pluggable Mechanisms — Extensibility

Each flow can use a different calculation mechanism. Adding new logic doesn't break existing code.

Strategy Pattern

// mechanisms/production.ts
export const production = (state, params) => {
  const efficiency = Math.min(
    params.semiconductor_reliability,
    params.battery_reliability
  ) / 100;
  
  return state.production_capacity * efficiency;
};

// mechanisms/market_share.ts
export const marketShare = (state, params) => {
  const fillRate = state.delivery / state.orders;
  return state.market_share * (1 + fillRate - 0.9);
};

How to Extend

Adding a New Mechanism

  1. Create new file: mechanisms/myMechanism.ts
  2. Export calculation function
  3. Register in mechanism registry
  4. Reference in JSON config

💎 No core engine changes needed — just add the mechanism and reference it in config!

⚖️ Trade-offs & Design Decisions

What We Chose

Decision Benefit
JSON config files Portable, version-controlled, AI-parseable
Client-side simulation Instant feedback, no server load
AI as suggestion layer Human stays in control
Pluggable mechanisms Easy to extend without refactoring

What We Traded Away

Trade-off Consequence
No database Manual file management for scenarios
Browser-only Large simulations may be slow
Proxy for AI Extra server component needed
Fixed Euler method May need RK4 for stiff systems
